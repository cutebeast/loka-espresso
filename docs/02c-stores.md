## QR Code Workflow

Tables use a **manual QR generation workflow** â€” QR codes are NOT auto-generated on table creation.

### Lifecycle

1. **PENDING** â€” Table created, no QR generated yet. Cannot be scanned.
2. **ACTIVE** â€” QR generated by service crew. QR code is printable and can be scanned.
3. **EXPIRED** â€” QR code past 30-minute expiry window. Must regenerate.
4. **IN USE** â€” Customer has scanned QR and placed a dine-in order.

### Workflow Steps

```
1. Service crew creates table â†’ Table is PENDING (no QR)
2. Service crew clicks "Generate QR" â†’ Table becomes ACTIVE
3. Service crew prints QR and places on table
4. Customer scans QR with PWA â†’ Places dine-in order
5. Order completed/cancelled â†’ Table released (IN USE â†’ ACTIVE)
```

### QR Code Expiry

- QR codes expire **30 minutes** after generation
- Service crew sees countdown timer on table card in admin dashboard
- Expired QR must be regenerated before customers can scan
- Regenerating invalidates the previous QR code (security measure)

### API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/admin/stores/{store_id}/tables` | Create table (no QR auto-generated) |
| POST | `/admin/stores/{store_id}/tables/{table_id}/generate-qr` | Generate QR for a PENDING table |
| GET | `/admin/stores/{store_id}/tables/{table_id}/qr-image` | Download QR as PNG |
| POST | `/admin/stores/{store_id}/tables/{table_id}/generate-qr` | Regenerate QR (invalidates old) |
| POST | `/tables/scan` | Customer scans QR to start dine-in order |
| POST | `/tables/{table_id}/release` | Release table after order completion |

### Security

- Each QR contains a cryptographically random `qr_token` (32-byte URL-safe token)
- Token validation uses constant-time comparison (`hmac.compare_digest`) to prevent timing attacks
- Tables without a QR token cannot be scanned â€” backend returns 403
- QR token is stored server-side only; never exposed to customers
